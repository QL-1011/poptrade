## 一、项目概述

### 1.1 项目背景

本系统是一个**轻量级电商单体后台系统**，面向中小型电商场景。系统包含**两套角色**：

- **管理员（后台）**：管理用户、分类、商品，处理订单状态流转
- **普通顾客（前台）**：注册登录、浏览商品、加入购物车、下单

两套角色共享同一个后端，通过 `role` 字段区分权限。系统刻意舍弃高并发、分布式、第三方支付对接等复杂能力，专注于**基础扎实、业务完整、代码规范**，适合面试口述和项目展示。

### 1.2 核心业务闭环

```
管理员端：  用户管理 ──→ 分类管理 ──→ 商品管理 ──→ 订单处理（发货等）
顾客端：    注册/登录 ──→ 浏览商品 ──→ 购物车 ──→ 下单 ──→ 查看订单
                                        ↑                    │
                                        └── 下单后清空 ──────┘
```

一个完整的顾客购物流程：

```
顾客注册 → 登录 → 浏览商品列表 → 按分类筛选 → 加入购物车
    → 修改购物车数量 → 提交订单 → 查看订单状态
    → 管理员后台发货 → 订单完成
```

对应的管理员操作流程：

```
管理员登录 → 创建商品分类 → 上架商品 → 查看订单列表
    → 对已付款订单发货 → 订单完成
```

### 1.3 项目目标

- 实现电商五大核心业务的完整闭环
- 覆盖 MyBatis 多表关联、Spring 事务控制、Redis 缓存、批量操作、状态流转等面试高频考点
- 提供标准 RESTful API，前后端分离架构
- 代码严格分层，结构清晰，易读易维护
- **明确区分管理员与顾客角色，权限清晰**

**架构说明：**
- **前后端分离**：后端只返回 JSON 数据，不渲染页面；前端可有管理后台 + 顾客端两个入口
- **单体架构**：所有模块在一个 SpringBoot 工程中，包结构分层
- **数据库 + 缓存**：MySQL 作为主存储，Redis 承担购物车和热点缓存
- **角色区分**：通过 Controller 路径前缀 + role 字段实现，不引入安全框架

#### 3.1.1 统一返回结果

```json
// 成功
{
  "code": 200,
  "message": "操作成功",
  "data": { ... }
}

// 失败
{
  "code": 500,
  "message": "系统异常，请联系管理员",

  "code": 400,
```

### 3.2 用户模块

**优先级：P0**

#### 3.2.1 角色定义

系统分为两种角色，在 `user` 表通过 `role` 字段区分：

| role 值 | 角色 | 能力范围 |
|---------|------|----------|
| 0 | 管理员 (ADMIN) | 管理用户、管理分类、管理商品、处理订单状态流转 |
| 1 | 普通顾客 (CUSTOMER) | 浏览商品、操作购物车、创建订单、查看自己的订单 |

**角色-权限矩阵：**

| 功能 | 管理员 | 顾客 | 说明 |
|------|--------|------|------|
| 用户管理（CRUD） | ✅ | ❌ | 管理员维护用户账号 |
| 用户注册 | ✅ | ✅ | 管理员可创建用户；顾客也可自行注册（注册时 role 默认为 1） |
| 用户登录 | ✅ | ✅ | 所有用户均可登录 |
| 分类管理 | ✅ | ❌ | 顾客只读（浏览商品时可看到分类） |
| 商品管理（增删改上下架） | ✅ | ❌ | 顾客只能浏览已上架商品 |
| 商品浏览 | ✅ | ✅ | 所有用户可查看 |
| 购物车操作 | ❌ | ✅ | 只有顾客有购物车 |
| 创建订单 | ❌ | ✅ | 只有顾客可以下单 |
| 订单查询 | ✅（全部） | ✅（仅自己的） | 管理员可查看所有订单，顾客只能看自己的 |
| 订单状态流转（发货等） | ✅ | ❌ | 管理员处理订单 |
| 取消订单 | ✅ | ✅ | 顾客可取消自己待付款/已付款的订单；管理员也可操作 |

| CT-05 | 查询购物车列表 | 展示当前用户购物车全部商品及数量 |

#### 3.5.2 Redis 数据结构设计

**Key格式：** `cart:{userId}`

**数据类型：** Hash

```
Key: cart:1001
Field-Value:
  product:1  →  3          （商品1，数量3）
  product:2  →  1          （商品2，数量1）
  product:5  →  2          （商品5，数量2）
```

**选择Hash的理由：**
- 一个用户一个购物车 Key，通过 Field 区分商品
- 支持原子性操作：hincrby 增减数量、hdel 删除单个商品、del 清空购物车
- 查询时 hgetall 获取全部数据，转为购物车列表

#### 3.5.3 购物车列表渲染

查询购物车时，需关联查询商品信息（名称、价格、图片）：

1. 从 Redis 获取 `cart:{userId}` 的全部数据（得到商品ID → 数量映射）
2. 批量查询 MySQL 商品表，获取商品详情
3. 组装返回购物车列表（商品信息 + 数量 + 小计）

#### 3.5.4 业务规则

| 规则编号 | 规则描述 |
|----------|----------|
| CTR-01 | 添加商品时，若数量 ≤ 0 视为非法请求 |
| CTR-02 | 修改数量为 0 时，等同于删除该商品 |
| CTR-03 | 购物车数据不存入MySQL，Redis即为唯一数据源 |
| CTR-04 | 提交订单后自动清空购物车（在订单模块事务中处理） |

### 3.6 订单模块

**优先级：P0（面试最重要模块）**

#### 3.6.1 功能列表

| 编号 | 功能 | 说明 |
|------|------|------|
| O-01 | 创建订单 | 校验库存 → 扣减库存 → 生成订单 → 生成订单详情 → 清空购物车（一个事务） |
| O-02 | 订单分页查询 | 支持按状态、时间区间筛选 |
| O-03 | 订单详情 | 含订单基本信息 + 订单商品明细 |

| O-05 | 取消订单 | 将订单状态改为"已取消"，恢复商品库存 |

#### 3.6.2 订单状态机

```
                    ┌──────────┐
                    │  已取消   │
                    └────▲─────┘
                         │ (待付款/已付款时可取消)
┌──────────┐      ┌──────┴──────┐      ┌──────────┐      ┌──────────┐
│  待付款   │ ──→  │   已付款    │ ──→  │  已发货   │ ──→  │  已完成   │
└──────────┘      └─────────────┘      └──────────┘      └──────────┘
```

**状态值定义：**

| 状态值 | 状态名 | 说明 |
|--------|--------|------|
| 0 | 待付款 | 订单刚创建 |
| 1 | 已付款 | 用户完成支付（模拟） |
| 2 | 已发货 | 商家已发货 |
| 3 | 已完成 | 订单完成（确认收货） |
| 4 | 已取消 | 订单取消 |

**合法状态转换：**

| 当前状态 | 可转换的目标状态 |
|----------|------------------|
| 待付款(0) | 已付款(1)、已取消(4) |
| 已付款(1) | 已发货(2)、已取消(4) |
| 已发货(2) | 已完成(3) |
| 已完成(3) | 不可变更 |
| 已取消(4) | 不可变更 |

#### 3.6.3 创建订单流程（面试核心）

```
beginTransaction {
    // 1. 遍历订单商品，逐一校验库存
    for each orderItem:
        if (product.stock < num):
            throw BusinessException("商品【xxx】库存不足")
    
    // 2. 扣减库存
    for each orderItem:
        productMapper.decreaseStock(productId, num)
    
    // 3. 生成订单记录（订单号、用户ID、总金额、状态=待付款）
    orderMapper.insert(order)
    
    // 4. 批量生成订单详情记录
    orderItemMapper.batchInsert(orderItems)
    
    // 5. 清空购物车
    redisTemplate.delete("cart:" + userId)
    
    // 6. 事务提交
} commitTransaction

// 异常回滚：库存恢复、订单不生成
```

**防止超卖：**
- 使用 `UPDATE product SET stock = stock - #{num} WHERE id = #{productId} AND stock >= #{num}`
- 检查受影响行数，若为 0 则说明库存不足（利用 MySQL 行锁 + 条件更新）

#### 3.6.4 业务规则

| 规则编号 | 规则描述 |
|----------|----------|
| OR-01 | 库存不足时拒绝创建订单，提示具体商品 |
| OR-02 | 创建订单必须在事务中完成，保证库存与订单数据一致性 |
| OR-03 | 订单号使用时间戳 + 随机数生成，保证唯一 |
| OR-04 | 取消订单时恢复库存 |
| OR-05 | 已完成和已取消的订单不可再变更状态 |
| OR-06 | 只有待付款和已付款状态可以取消 |

## 五、缓存设计

### 5.1 Redis数据结构汇总

| 场景 | Key格式 | 数据类型 | 过期策略 |
|------|---------|----------|----------|
| 购物车 | `cart:{userId}` | Hash | 不过期（提交订单后主动删除） |
| 热门商品列表 | `product:hot` | String (JSON) | TTL 30分钟 |
| 商品详情 | `product:{id}` | String (JSON) | TTL 1小时 |

### 5.2 缓存策略

```
查询热门商品:
    data = Redis.get("product:hot")
    if data != null:
        return data
    else:
        data = DB.query("SELECT ... ORDER BY sales DESC LIMIT 10")
        Redis.set("product:hot", JSON(data), 30min)
        return data

更新商品:
    DB.update(product)
    Redis.del("product:" + productId)     // 删除商品详情缓存
    Redis.del("product:hot")              // 删除热门商品缓存（可能影响排名）
```

### 5.3 缓存一致性

| 操作 | 缓存处理 |
|------|----------|
| 新增商品 | 删除 `product:hot` |
| 修改商品 | 删除 `product:{id}` 和 `product:hot` |
| 删除商品 | 删除 `product:{id}` 和 `product:hot` |
| 创建订单（影响库存和热度） | 删除 `product:{id}` 和 `product:hot` |

采用 **先更新DB，再删除缓存** 的策略，避免并发场景下的数据不一致。
